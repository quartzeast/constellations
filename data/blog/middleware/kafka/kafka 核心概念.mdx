## 主题、分区和偏移

### 主题（Topic）

![Kafka](https://cdn.jsdelivr.net/gh/quartzeast/constellations-blog-images@main/img/2024/09/21/2024:09:21-13:00:14:932.webp)

Kafka 主题（Topic）是 Kafka 集群中的特定数据流。一个 Kafka 集群可以有很多主题，比如 `logs`, `purchases`, `twitter_tweets`, `trucks_gps` 等。Kafka 中的主题其实就是数据流。若与数据库类比，Kafka 主题类似于数据库中的表，但 Kafka 主题没有任何约束（Constraints）。可以将任何内容发送到 Kafka 主题，且不用对数据进行验证。我们能在 Kafka 集群中创建任意数量的主题。

Kafka 集群中的主题是靠名称进行区分的。 `logs`, `purchases`, `twitter_tweets`, `trucks_gps` 这些都是主题的名称。Kafka 主题支持任意类型的消息格式，比如 JSON、Avro、文本文件、二进制文件等。主题中的消息序列被称作数据流（data stream），这也是 Kafka 被称为数据流平台的原因，因为你能通过主题创建数据流。

需要注意，Kafka 主题与数据库表类似，不过无法对主题进行查询。若要向 Kafka 主题添加数据，需使用 Kafka 生产者；若要从主题读取数据，则需使用 Kafka 消费者。但 Kafka 本身不具备查询功能。

关于 Kafka 主题的小结：

- 主题（Topic）：一个特定数据流
- 像数据库中的表（但没有任何约束）
- 可以创建任意数量的主题
- 一个主题通过其名称（name）来标识
- 支持任意类型的消息格式
- 消息的序列称为一个数据流（data stream）
- 不能对主题进行查询，必须使用 Kafka 生产者（Producer）发送数据，使用 Kafka 消费者读取数据。

### 分区（Partition）和偏移（Offset）

主题可进一步划分为分区。一个主题可由多个分区组成，比如 100 个分区。在下图示例，创建了一个包含 3 个分区的 Kafka 主题，即分区 0、分区 1 和分区 2。

![image-20240921140528847](https://cdn.jsdelivr.net/gh/quartzeast/constellations-blog-images@main/img/2024/09/21/2024:09:21-14:05:33:670.png)

发送到 Kafka 主题的消息最终会进入这些分区。每个分区内的消息都是有序的。例如，发送到分区 0 的第一条消息拥有 ID `0`，`1`，`2`... `9`。随着不断向分区写入消息，这个 ID 会不断增加。同样，当向 Kafka 主题的分区 1 写入数据时，ID 也会会不断增加。

在这些分区中写入的消息都会获得一个从零开始递增的 ID，这个 ID 被称为 Kafka 分区偏移（Kafka partition offset）。

每个分区都有不同的偏移量。Kafka 主题也是不可变的，这意味着一旦数据写入分区，就无法更改。因此，我们无法在 Kafka 中删除数据，也无法更新数据，只能不断地向分区写入数据。

小结：

- 主题被划分为分区（如 100 个分区）
  - 每个分区内的消息都是有序的
  - 每个分区内的消息都会获得一个增加的 id，叫做偏移
- Kafka 主题是不可变的（immutable）：一旦数据写入分区，它就不能再被改变

### 主题示例：`truck_gps`

![image-20240921141426343](https://cdn.jsdelivr.net/gh/quartzeast/constellations-blog-images@main/img/2024/09/21/2024:09:21-14:14:30:227.png)

我们以 `trucks_gps` 为例进行具体说明。假设你拥有一支卡车车队，每辆卡车都配备一个 GPS。GPS 会将卡车的位置报告给 Kafka。每辆卡车每 20 秒向 Kafka 发送一条消息，消息中包含一些信息，比如卡车 ID 和卡车位置（如纬度和经度）。

因此这些卡车是数据生产者，它们会向一个名为 `trucks_gps` 的 Kafka 主题发送数据，这个主题将包含所有卡车的位置。

于是，我们将数据发送到 `trucks_gps` 主题中。因为这个主题是由多个分区组成的，如前所述，我们选择创建一个包含 10 个分区的主题。

一旦在 Kafka 中创建了这个主题，我们便有了一个应用场景。例如，我们希望有消费者来消费 `trucks_gps` 数据，并将其发送到位置 DashBoard 上，以便实时跟踪所有卡车的位置。或者，我们可能希望有一个通知服务来消费同样的数据流，该通知服务可在卡车即将送达时向客户发送通知。

这正是 Kafka 非常有用的原因，因为多个服务可以从同一个数据流（主题）中读取数据。

### 主题、分区和偏移的注意事项

![image-20240921140528847](https://cdn.jsdelivr.net/gh/quartzeast/constellations-blog-images@main/img/2024/09/21/2024:09:21-14:05:33:670.png)

- 一旦数据写入一个分区，就不能再改变它，这就是所谓的不可变性（immutability）
- Kafka 中的数据只会被保留一段有限的时间，默认为一周，但这个时间可以配置。也就说，超过一周后，数据会被删除
- 偏移量只对特定的分区有意义
  - 偏移量在各个分区之间是重复的。分区 0 中的偏移量 3 代表一条消息，但它与分区 1 中的偏移量 3 所代表的数据不同
  - 即使之前的消息已被删除，偏移量也不会被重新使用，它会随着你发送到 Kafka 主题中的消息而逐步增加
- 消息的顺序仅在各个分区内部是有保证的，而跨分区则无法保证消息的顺序（如果需要保证顺序，我们需要想其他办法来实现）
- 当数据发送到 Kafka 主题时，默认会分配到一个随机的分区。除非提供一个键（key）。
- 在 Kafka 主题可以根据需要设置任意数量的分区

## 生产者和消息 Key

### 生产者（Producer）

![image-20240921142812933](https://cdn.jsdelivr.net/gh/quartzeast/constellations-blog-images@main/img/2024/09/21/2024:09:21-14:28:16:089.png)

主题（topic）存储数据，但要向主题写入数据，我们需要使用 Kafka 生产者。生产者会向主题的分区写入数据。如上图中的主题 Topic A 有分区 0、分区 1、分区 2，数据写入时会伴随着偏移量的增加顺序进行。

生产者将数据发送到 Kafka 主题的分区中。生产者需要事先知道要将数据写入哪个分区，以及该分区所在的 Kafka 服务器（也就是 Kafka broker）。生产者事先知道消息将被写入哪个分区。生产者在写入数据时会提前决定要写入哪个分区。

在 Kafka 服务器分区故障的情况下，生产者知道如何进行自动恢复（这里是指自动可以将消息发送到另一个没有故障的服务器的分区上。发送数据时会进行负载均衡，生产者根据某种机制可以将数据发送到所有分区（这里不是指将数据一次性发送到多个分区，而是指有可以向多个分区发送数据的能力），这也是 Kafka 能够实现扩展的原因，即主题内有多个分区，每个分区可以接收来自一个或多个生产者的消息。

小结：

- 生产者将数据写入到主题（主题由分区组成）
- 生产者知道写入到哪个分区（和哪个 Kafka broker）
- 为了避免 Kafka broker 故障，生产者将自动恢复

### 生产者：消息 key

![image-20240921143547088](https://cdn.jsdelivr.net/gh/quartzeast/constellations-blog-images@main/img/2024/09/21/2024:09:21-14:35:50:698.png)

生产者（producers）有消息键（message keys）和消息（messages）。消息本身包含数据，但我们可以添加一个可选的 key，这个 key 可以是任何东西：可以是字符串、数字、二进制等。在上图中，有一个生产者正在向一个有两个分区的主题写入数据。如果键是空（null），那么数据会被轮流发送，意味着它会依次发送到分区 0、分区 1、分区 0，以此类推，这就是实现负载均衡的方式。键等于 `null` 意味着在生产者消息中未提供键。

但是如果键不是空（`null`），那么键就有某个值；它可以是字符串、数字、二进制等。Kafka 生产者有一个非常重要的特性：所有具有相同键的消息将始终被写入到同一个分区，这得益于一种哈希策略。

因此，当我们指定一个键时，这意味着我们需要根据特定字段进行消息排序。还记得之前关于卡车的例子吗？我们有许多卡车，我们希望能够按照每辆卡车的位置顺序获取数据。在这种情况下，我会将卡车 ID 作为消息的键。例如卡车 ID `123`，它总是会被发送到分区 `0`，这样我就可以按照卡车 ID `123` 的顺序读取数据。而卡车 ID `234` 也总是会被发送到分区 `0`，哪个键最终在哪个分区中，这是由哈希算法决定的。例如，另一个卡车 ID，比如 `345` 或 `456`，将总是会被发送到主题 `A` 的分区 `1`。

小结：

- 生产者可以选择和消息一起返送一个 key（字符串、数字、二进制等）
- 如果 `key=null`，数据以轮流（round robin）的方式发送
- 如果 `key!=null`，相同 key 的所有消息将始终发送到同一分区（通过哈希算法）
- 如果要求消息是有序的，通常会一起发送一个 key

### Kafka 消息解构

![image-20240921144359436](https://cdn.jsdelivr.net/gh/quartzeast/constellations-blog-images@main/img/2024/09/21/2024:09:21-14:44:02:585.png)

消息键（key）是 Kafka 消息的一部分，上图是生产者创建的消息的结构。键（key）可以为空（null），并且以二进制格式表示。然后是值（value），这是消息的内容，也可以为空，但通常情况下不会为空，而是会包含消息的实际值。

我们可以为消息添加压缩机制。如果希望消息更小，可以指定压缩机制，例如 gzip、Snappy、LZ4 或 zstd。然后，我们还可以为消息添加 Header，这是一个可选的键值对列表。接着是消息将要发送到的分区，以及偏移量。最后，是一个时间戳，可以由系统自动设置，也可以由用户指定。这就是 Kafka 消息的结构，它会被发送到 Kafka 进行存储。

### Kafka 消息序列化器

### ![Kafka](https://cdn.jsdelivr.net/gh/quartzeast/constellations-blog-images@main/img/2024/09/21/2024:09:21-15:04:40:847.webp)

在 Kafka 中，消息是如何创建的呢？这涉及到 Kafka 消息序列化器（Message Serializer）。因为 Kafka 是一种非常优秀的技术，它只接受来自生产者的字节序列作为输入，并将字节序列作为输出发送给消费者。但是当我们构造消息时，它们并不是字节，所以需要进行消息序列化。序列化意味着将你的数据、对象转换为字节流。

这些序列化器只会用于键和值。举个例子，假设我们有一个键对象，比如车辆 ID `123`，然后值就是一个字符串，`"hello world"`。在编程语言中，这些并不是字节流，而是对象（值）。然后我们指定键的序列化器为整数序列化器。Kafka 生产者足够智能，能够将这个键对象 `123` 通过序列化器转换成一系列字节，从而得到键的二进制表示。对于值对象，我们指定一个字符串序列化器，在这种情况下，值和键的序列化器是不同的。这意味着 Kafka 生产者会智能地将字符串 `"hello world"` 转换成一系列字节流作为值。现在，键和值都已经转换成了二进制表示，这个消息现在就准备好发送到 Kafka 中了。

Kafka 生产者附带了常见的序列化器，帮助你进行这种转换。我们有字符串序列化器，包括字符串的 JSON 表示，整数序列化器，浮点数序列化器，Avro 序列化器，Protobuf 序列化器等等。我们可以在很多消息序列化器中找到合适的工具。

小结：

- Kafka 从生产者只接受字节作为输入并且将字节作为输出发送给消费者
- 消息序列化表示将对象/数据转换为字节
- Kafka 消息序列化器在值和 key 上使用
- 常用的序列化器：
  - String（包括 JSON）
  - Int, Float
  - Avro
  - Protobuf

### Kafka 消息 key 哈希

![image-20240921151559571](https://cdn.jsdelivr.net/gh/quartzeast/constellations-blog-images@main/img/2024/09/21/2024:09:21-15:16:02:500.png)

消息 Key 是如何被哈希的呢？有一个叫做 Kafka Partitioner 的东西，它是一种代码逻辑，用来接收记录，并决定将其发送到哪个分区。当我们发送消息时，生产者的分区逻辑会检查这条消息，并决定其发送的分区，然后生产者将其发送到 Kafka。Key 的哈希过程用于确定将键映射到分区的方式，在默认的 Kafka Partitioner 中，键会使用 Murmur2 算法进行哈希。

```shell
targetPartition = Math.abs(Utils.murmur2(keyBytes)) % (numPartitions - 1)
```

以上是目标分区的计算公式，由公式可知，生产者会对键的字节流应用 Murmur2 算法，然后确定目标分区。因此生产者是通过哈希 Key 来决定消息最终存储在哪个分区的。

小结：

- Kafka partitioner 是一种代码逻辑，接收一条记录并决定将其发送到哪个分区去
- Key Hashing 是决定将 key 映射到哪个分区的过程
- 在默认的 Kafka partitioner 中，是使用以上的公式，应用 murmur2 算法对 key 进行哈希的

## 消费者和反序列化

我们已经学习了如何向主题中生产数据。现在，若要从主题中读取数据，就需要用到消费者。消费者实现了拉取（pull）模型，即消费者会向 Kafka 服务器请求数据，之后等待服务器响应。并非由 Kafka 服务器推送（push）数据给消费者，而是消费者主动拉取数据。

我们以一个包含三个分区的主题的示例来说明。一个消费者可能希望从主题 A 的分区 0 中读取数据，并按照偏移量顺序读取数据。另一个消费者可能选择从主题的多个分区读取数据，比如从分区 1 和分区 2 中读取数据。当消费者需要从分区读取数据时，它们会自动知道从哪个 Kafka 服务器读取数据。如果一个服务器故障，消费者也会智能地进行故障恢复。

从所有这些分区读取的数据会按照偏移量从低到高的顺序读取，比如分区 0 的偏移量从 0 到 11。每个消费者都会按照这个顺序读取数据。但要记住，跨分区之间并没有顺序保证，因为它们属于不同的分区，唯一的顺序保证是在每个分区内部。

消费者读取消息后，需要将从 Kafka 接收的字节转换为对象或数据。消息的键和值都是以二进制格式存储的，对应于 Kafka 消息中的数据。然后我们需要将它们转换成我们编程语言可以使用的对象。

因此，消费者需要预先知道消息的格式是什么样的。在这个例子中，消费者知道键是整数类型，因此会使用整数反序列化器将接收到的字节转换为整数。然后键对象就会恢复成 123 这样的整数。对于值，我们知道需要一个字符串类型的反序列化器，因为我们期望 Kafka 主题中的值是字符串类型，因此这个反序列化器将把字节流转换为字符串，从而恢复出值对象 "hello world"。
